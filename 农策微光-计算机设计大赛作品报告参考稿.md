## 使用说明

### 先说明一个关键事实

你们目前拿到的 `05-3 作品报告（人工智能实践赛，2026版）模板.docx`，正文实际更接近一份**作品信息概要表/提交信息表**，并不是传统意义上带固定章节的长篇设计报告模板。

因此，本参考稿采用“双层结构”：

1. **Part A：作品信息概要表填写参考**

   - 目标是“拿来就能填表”
2. **Part B：长篇作品报告正文参考**
   - 对应后续要交给组员分工扩写的完整正文
   - 目标是“拿来就能写长文、做 PPT、录答辩视频”

### 标记规范

- `【可直接使用】`：可以基本原文复制
- `【建议改写后使用】`：建议结合你们真实截图、成员、演示数据做小幅修改
- `【待补充】`：必须由你们自己补充的信息

### 本参考稿采用的统一口径

- 作品名称统一写为：**农策微光——面向农业经营主体的政策智能匹配与申报辅助平台**

- 不夸大成“已大规模商用落地”，统一写成：**课程项目 / 竞赛作品 / 面向真实场景的原型系统**

## 第一章 作品概述

### 1.1 项目背景

### 【可直接使用】参考正文

```text
农业政策是农业经营主体获取补贴、参与项目申报、提升经营能力的重要依据，但在实际应用中，政策文本往往存在来源分散、篇幅冗长、术语专业、门槛复杂等问题。对于农户、家庭农场、农民合作社和农业企业而言，真正困难的不是“知道有政策”，而是“看不懂政策”“判断不了自己是否符合”“不知道下一步该准备什么材料”。这类问题长期依赖人工咨询、关键词检索和经验判断，不仅效率低，而且容易遗漏重要条款。

基于这一现实痛点，我们设计并实现了“农策微光”系统，希望通过结构化规则建模、画像驱动匹配、大模型增强解读和后台治理闭环，将原本分散、抽象、难执行的农业政策信息转化为可理解、可判断、可行动的辅助服务。
```

### 1.2 真实痛点

建议从以下四个角度展开：

- 政策查找难
- 政策理解难
- 条件比对难
- 申报准备难

### 1.3 目标用户

【可直接使用】写法：

- 普通用户：农户、家庭农场、合作社、农业企业等农业经营主体
- 管理用户：政策维护人员、管理员、政策库运营人员

### 1.4 作品定位

【可直接使用】参考表述：

```text
本作品定位为面向农业经营主体的政策智能匹配与申报辅助平台。它既不是单纯的政策搜索工具，也不是纯问答机器人，而是一个以结构化政策规则为核心、以大模型增强解释为特色、以后台治理链路为支撑的一体化农业政策服务系统。
```

### 1.5 一句话价值主张

【可直接使用】候选：

- 让农业经营主体从“看政策”走向“懂政策、判政策、用政策”
- 让合适的政策主动找到合适的人
- 让政策申报从经验驱动转向规则与智能协同驱动

## 第二章 问题分析与需求来源

### 2.1 现实问题分析

建议围绕以下逻辑组织：

1. 公开政策虽然丰富，但入口分散
2. 原文规范性强，普通用户阅读成本高
3. 政策适用条件往往涉及主体、区域、面积、资质、时间等多维约束
4. 用户缺少一个统一的判断和行动辅助平台

### 2.2 传统方案的局限性

【可直接使用】参考正文

```text
传统政策服务方式主要依赖关键词检索、人工阅读和经验咨询。这种方式在面对结构复杂、条件交叉、适用范围动态变化的农业政策时，难以快速形成针对某个经营主体的明确判断。即便用户找到了相关政策，也往往无法在短时间内回答“是否适用”“缺少什么条件”“下一步该做什么”等更具体的问题。
```

### 2.3 需求来源口径

【建议改写后使用】参考表述：

```text
本项目需求来源于真实农业政策服务场景与课程项目实践任务。一方面，团队关注到农业经营主体在政策获取、理解与申报过程中的长期痛点；另一方面，项目组结合经济管理学院相关调研背景与已有课程实践，逐步明确了“画像建模、规则匹配、白话解读、后台治理、趋势展示”五条主线，形成了本作品的核心需求框架。
```

### 2.4 为什么做“画像 + 规则 + 解读 + 治理”一体化平台

这节建议写成四段：

- 只有画像，没有规则，无法精确匹配
- 只有规则，没有解读，普通用户仍然难理解
- 只有解读，没有治理，政策库无法持续扩充
- 只有前台，没有后台，系统难以长期维护和演示

## 第六章 人工智能能力设计

### 6.1 DeepSeek 在项目中的作用

【可直接使用】参考正文

```text
本作品将 DeepSeek 作为人工智能增强能力接入系统，但并未将其作为唯一决策核心，而是把其置于结构化生成、摘要解释和趋势报告等适合生成式模型发挥优势的环节。具体而言，DeepSeek 主要承担政策文本编译、条件树生成、审核辅助建议、政策风向标报告生成以及对话增强等任务。
```

### 6.2 为什么不是纯大模型方案

【可直接使用】参考正文

```text
如果完全依赖大模型直接回答“用户是否符合政策”，虽然可以快速生成自然语言，但结果可追溯性、稳定性和确定性较弱。为此，本作品采用“规则匹配为主、AI 增强为辅”的策略：匹配结果由本地规则引擎根据结构化条件树执行确定性判断，AI 主要负责从原文中提取结构、生成摘要、提供审核建议和增强解释，从而兼顾了准确性、可解释性与交互体验。
```

### 6.3 条件树机制

这部分必须重点写，建议包含：

- 条件树是政策结构化的核心中间层
- 根节点 / 分组节点 / 条件节点
- 支持 must / should
- 支持主体、区域、时间、资质、规模、门槛等字段
- 匹配引擎基于条件树执行判断

### 【可直接使用】条件树说明段

```text
条件树机制是本作品区别于普通问答系统的关键设计。系统先将政策原文编译为结构化条件树，再由本地规则引擎对经营主体画像进行逐项判断。条件树不仅描述“是否匹配”，还保留“为什么匹配/不匹配”的节点依据，使系统能够给出清晰的缺口分析、风险提示与行动建议。
```

### 6.4 AI 合规使用说明

【可直接使用】参考文本

```text
本作品中的 AI 能力主要用于文本结构化、摘要生成、审核辅助和趋势报告生成，不直接替代规则引擎做最终申报资格判断。涉及用户画像与政策匹配的关键结论，仍由本地规则逻辑完成，以保证结果的确定性和可解释性。AI 生成内容仅作为增强与辅助，不作为不可追溯的唯一依据。
```
