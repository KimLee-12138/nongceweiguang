***\*1 引言\****

***\*1.1 编写目的\****

本说明书用于描述“农策微光”系统的业务目标、功能需求、数据需求、性能需求、运行需求及其他质量要求，为课程项目开发、阶段验收、演示答辩、后续维护与功能扩展提供统一依据。

本说明书面向以下读者：项目组成员，用于统一需求理解与开发边界。指导教师与课程评审人员，用于了解系统的业务定位、功能完成度和实现范围。后续维护人员，用于快速理解系统结构、接口关系与核心数据对象。

***\*1.2 背景\****

在农业政策申报场景中，用户通常面临以下问题：政策来源分散、原文专业性强、适用条件复杂、个人或主体信息与政策条款难以快速对齐。传统方式往往依赖人工查阅政策文件、逐条比对条件，效率较低，且容易遗漏关键条款。

“农策微光”面向这一痛点，尝试构建一个农业政策智能匹配与申报辅助平台。系统围绕“农户画像挂载、政策规则化、确定性匹配、白话解读、后台治理”五条主线展开，将政策原文转化为结构化条件树，并结合农户或农业经营主体画像输出更可解释的匹配结果和行动建议。

本项目当前采用前后端分离架构实现，前端负责页面交互与结果展示，后端负责数据存储、政策管理、规则匹配、对话接口、文件解析、爬虫同步和风向标生成等能力。

***\*2 任务概述\****

***\*2.1 项目概述\****

***\*2.1.1 项目来源及背景\****

本项目来源于经济管理学院的真实场景调研，目标是围绕真实农业政策服务场景，设计并实现一套具有实际演示价值的农业政策智能匹配与辅助申报系统。

从业务背景看，农业经营主体在寻找适合自身情况的扶持政策时，往往需要面对政策原文长、门槛条款多、适用范围复杂、资料准备要求不明确等问题。系统需要提供一种比传统关键词搜索更直观、更可解释的解决方案，将“政策可看见”进一步转化为“政策可理解、可判断、可行动”。

***\*2.1.2 项目目标\****

***\*（1）提供专业的农业政策智能匹配与申报辅助平台：\****打造一个集成农户画像管理、政策智能匹配、对话式政策解读、政策洞察展示、政策风向标生成、政策库管理、文件批量导入、爬虫自动抓取等功能的专业化农业政策服务平台，使用户可以在同一个平台上完成农业政策查询、匹配、解读与申报准备等全流程操作。

***\*（2）提升农业经营主体政策申报的便利性：\****通过系统提供的智能匹配、白话解读、缺口分析与行动建议等功能，帮助农户、家庭农场、农业合作社、农业企业等经营主体快速理解政策、精准匹配政策，降低政策申报门槛与时间成本。

***\*（3）优化政策信息获取与条件比对效率：\****通过政策结构化编译、条件树规则匹配、农户画像自动对齐等功能，帮助用户高效筛选适配政策，解决政策分散、条款难懂、人工比对效率低等痛点。

***\*（4）提高政策管理与维护的效率和效益：\****通过管理员后台、文件解析、手动 / 自动爬虫、政策编译等功能，帮助政策维护人员快速扩充与更新政策库，提升政策数据治理效率。

***\*（5）提供政策趋势洞察与决策辅助：\****通过政策风向标、智库词典、政策洞察数据展示等功能，为用户与管理者提供农业政策趋势分析与机会信号，辅助农业经营与政策管理决策。

***\*（6）提供稳定可靠的系统支撑能力：\****通过前后端分离架构、数据库持久化、接口鉴权、外部服务兼容等设计，保障系统稳定运行、数据安全可追溯，支持课程演示与后续扩展。

***\*（7）功能完善和扩展性提升：\****持续优化核心匹配与解读功能，支持新增政策数据源、扩展画像字段、迭代规则匹配算法，满足农业政策场景不断变化的需求，提升系统功能完善度与扩展性。

***\*2.1.3 系统功能概述\****

***\*（1）农户画像管理：\****支持用户创建、编辑、查看自身农业经营主体画像，包含主体类型、经营面积、绿色认证、灌溉方式及扩展字段等信息，作为政策匹配的基础数据。

***\*（2）政策智能匹配：\****基于用户画像与结构化政策条件树进行规则化匹配，自动判断申报资格，输出匹配结果、缺口分析与行动建议。

***\*（3）对话式政策解读：\****以聊天交互方式为用户提供政策白话解读、条件说明、申报指导，支持流式响应与政策体检结果展示。

***\*（4）相关政策推荐：\****根据用户画像批量筛选政策库，按匹配程度排序推荐适配政策，支持快速浏览与换一批刷新。

***\*（5）政策洞察展示：\****聚合政策数据生成可视化统计图表，展示政策数量、来源分布、受众分布与趋势概况，便于用户直观了解政策整体情况。

***\*（6）管理员登录与鉴权：\****提供管理员账号登录、身份校验与令牌管理，保障后台操作安全。

***\*（7）结构化政策管理：\****支持管理员对政策进行新增、编辑、删除、列表查询，实现政策库规范化管理。

***\*（8）政策文本编译：\****将政策原文自动解析为结构化条件树，生成可计算、可匹配的规则化政策对象。

***\*（9）文件解析与批量导入：\****支持上传 Word、PDF 文件，提取文本并完成 OCR 识别，实现政策批量编译与入库。

***\*（10）手动与自动爬虫导入：\****支持手动抓取政策数据，可配置自动爬虫周期与参数，实现政策库持续更新与扩充。

***\*（11）政策风向标与智库词典：\****定期生成政策趋势周报、机会信号与术语解释，辅助用户理解政策方向与重点内容。

***\*2.2 用户特点\****

本系统主要面向以下两类用户：

（1）普通用户：包括农户个人、家庭农场、农民合作社、农业企业等农业经营主体。其特点是

· 了解政策需求强，但缺乏对政策条文的专业理解能力。

· 更关注“我能不能申报”“我还缺什么条件”“接下来该做什么”。

· 期望使用门槛低，偏好可视化、白话化、对话式交互。

（2）管理用户：即系统管理员或政策库维护人员。其特点是

· 需要维护政策知识库，保证政策条目持续入库和更新。

· 需要管理自动爬虫与文件导入等后台流程。

· 需要对外展示系统运行成果和平台数据概况。

***\*2.3 假定和约束\****

***\*（1）开发期限约束：\****项目需在软件工程课程结束前完成全部开发、测试与试点部署工作，交付可运行版本及相关成果文档。

***\*（2）第三方API支持：\****假定第三方模型服务在需要时可用，否则部分 AI 相关功能仅返回提示或降级结果。

**（3）用户体验**：软件需要提供良好的用户体验，包括界面设计、操作流畅性等方面，以提升用户满意度和使用体验。

***\*（4）法律法规约束：\****本项目定向爬虫采用低频礼貌抓取策略，严格遵守政府网站 robots 协议；同时，系统全面接入国内合规的商业大模型（如 DeepSeek、通义千问等），确保政务数据处理的绝对安全与自主可控，同时，针对调研与系统收集的农户画像数据，实行严格的脱敏与本地化存储，切实保障涉农主体的数据隐私。

**（5）开发团队**： 本平台开发团队成员五人团队成员数量与开发周期相对于专业的项目开发团队的配置还有不小的差距，并且作为在校学生许多开发中需要用到的知识还未充分掌握，所以开发得到的成果于市场广泛运用的软件存在较大差距。

***\*3.2 功能描述\****

***\*3.2.1 农户画像管理\****

功能说明：系统支持用户创建、查看与修改农户或农业经营主体的画像信息，画像数据将作为政策匹配的核心依据。用户可录入主体名称、主体类型、经营面积、绿色认证情况、灌溉方式等基础信息，也可根据实际需求补充作物类型、经营年限、保险情况等动态扩展字段。系统会对基础字段进行结构化存储，对不固定的扩展字段采用键值对方式灵活保存，最终向用户返回画像主键 ID、完整的画像详情以及该画像对应的历史政策匹配记录。

***\*3.2.2 政策智能匹配\****

功能说明：系统能够依据已保存的农户画像与结构化政策条件树自动开展规则匹配，精准判断用户是否满足对应政策的申报要求，并输出完整的结构化匹配结果。匹配过程中，系统会基于政策条件树执行本地确定性判断，清晰区分必须满足条件与建议满足条件，同时自动汇总匹配失败节点、具体原因、相关风险提示与可执行行动建议。最终输出内容包括是否完全匹配、未通过的必须条件与建议条件、全部条件节点的评估结果，以及通俗易懂的白话解释、清晰的行动清单和必要的风险预警。

***\*3.2.3 对话式政策解读\****

功能说明：用户可在聊天工作台中直接发起政策咨询，系统会以流式文本形式返回自然语言回答，并可同步附带政策体检结果与专业解读内容。用户在发起咨询时可输入问题内容，也可选择绑定画像 ID、指定政策 ID 或政策原文 ID，同时切换匹配模式、解读模式等不同聊天模式。系统会根据用户问题自动检索相关政策内容，在具备有效画像的情况下结合政策规则生成政策体检结果，并通过 SSE 方式持续推送回答内容。最终为用户返回流式文本回答、完整的政策体检结果对象，以及清晰易懂的政策解释或操作引导信息。

***\*3.2.4 相关推荐政策\****

功能说明：系统会根据用户当前使用的画像，对政策库进行批量匹配预览，筛选并返回一组适合该经营主体的政策候选。用户只需提供画像 ID、偏移量与需返回的条数，系统即可对全量政策库进行预览匹配，自动过滤公示类及不适合申报的政策条目，再按照匹配程度高低进行排序。最终向用户展示政策 ID、标题、来源、摘要、原文链接，同时标注是否完全匹配、是否包含风险提醒以及是否提供行动建议。

***\*3.2.5 政策洞察展示\****

功能说明：系统提供专门的政策洞察页面，用于展示政策数量、来源分布、目标受众、趋势分析等辅助信息，无需用户手动输入业务参数，页面加载时会自动请求数据快照。后端会自动聚合洞察数据并进行缓存，前端接收数据后以图表与指标卡片形式直观展示。最终输出内容包含政策核心 KPI 汇总、趋势分析数据、来源分布数据以及受众分布数据，帮助用户全面了解政策整体情况。

***\*3.2.6 管理员登录与鉴权\****

功能说明：系统支持管理员使用账号密码登录，并通过 JWT 令牌对后台所有接口进行安全保护。管理员输入用户名与密码后，系统会自动校验账号密码的合法性，校验通过则签发有效访问令牌，所有后台受保护接口均通过 Bearer Token 完成身份校验。登录成功后，系统向管理员返回有效访问令牌与当前管理员的完整身份信息，确保后台操作安全可控。

***\*3.2.7 结构化政策管理\****

功能说明：系统支持管理员对结构化政策执行新增、列表查看、详情查看、编辑与删除等全流程管理操作。管理员可录入政策标题、来源、版本、摘要、原文链接与结构化条件树等信息，系统将政策以结构化对象形式保存至数据库，支持分页查询与按 ID 精准查询，删除政策时会同步清理相关匹配记录的关联引用，保证数据一致性。最终输出政策列表、单条政策详情以及新增、更新、删除操作的执行结果。

***\*3.2.8 政策文本编译\****

功能说明：系统支持将原始政策文本提交至规则编译服务，自动生成可用于匹配的结构化政策对象。管理员只需提供政策标题、政策来源与原始政策文本，系统会调用大模型对文本进行结构化解析，按照系统规则生成标准条件树与摘要信息。操作完成后，系统返回编译完成的结构化政策对象，若编译过程出现异常则返回明确的错误提示信息。

***\*3.2.9 文件解析与批量导入\****

功能说明：系统支持管理员上传 Word 或 PDF 格式文件，自动提取正文内容后由管理员选择条目，批量导入至结构化政策库。管理员上传.docx 或.pdf 文件并选定需导入的条目后，系统会自动提取文件正文，针对扫描版 PDF 按需调用 OCR 识别，将解析结果返回前端供人工勾选确认，再对选中条目执行批量编译与入库操作。最终输出文件解析结果列表、批量导入成功与失败数量，以及详细的失败明细信息。

***\*3.2.10 手动政策爬虫导入\****

功能说明：系统支持管理员从已对接的数据源手动抓取政策原文，经复核确认后导入政策库。管理员指定数据源标识与抓取页数等参数后，系统自动爬取对应数据源的列表页与详情页内容，完成去重、正文抽取、原文链接整理等处理，生成可导入的政策候选条目。最终返回本次抓取的政策候选列表，包含原文标题、来源、正文及链接等完整信息。

***\*3.2.11 自动爬虫配置与运行记录\****

功能说明：系统支持管理员配置自动爬虫的运行参数，包括开关状态、数据源、抓取间隔、最大抓取页数，并可随时查看最近运行情况。管理员设置是否启用自动爬虫、指定数据源列表、设置间隔小时数与单次抓取页数上限后，系统将配置信息持久化保存至系统配置表，定时任务按设定周期自动执行抓取与导入操作，并完整记录运行结果。最终输出当前爬虫配置内容、最近一次运行状态，以及抓取、筛选、导入、失败的数量统计。

***\*3.2.12 政策风向标与智库词典\****

功能说明：系统支持自动或手动生成政策风向标周报与智库词典，帮助用户快速理解近期政策趋势。可由定时任务自动触发或管理员手动触发，系统基于后端结构化政策信号与近期政策数据，聚合关键政策信号，调用大模型生成报告摘要与正文内容，同步生成同周期的术语解释列表。最终输出政策总览、主题趋势、机会信号，展示最新风向标报告列表与详情，以及完整的智库词典条目列表。

***\*4.4.1数据库类型\****

系统采用 MySQL 作为关系型数据库，后端通过 SQLAlchemy ORM 对数据表进行映射与访问。

***\*4.4.2主要表之间的关系\****

· user_profiles 与 match_records 为一对多关系，即一个画像可对应多条匹配记录。

· policies 与 match_records 为一对多关系，即一条政策可被多个画像匹配。

· system_config 采用键值对方式存储全局配置。

· compass_reports 与 compass_glossary 通过发布时间形成逻辑上的周期对应关系。

***\*4.4.3主要设计特点\****

· policies.condition_tree 使用 JSON 存储完整条件树，便于直接参与规则匹配。

· user_profiles.extra_data 使用 JSON 存储动态扩展字段，便于适应不同政策场景。

· match_records.match_detail 使用 JSON 存储完整匹配结果，便于后续展示与复盘。

· hubei_policies_raw.full_text 存储原文全文，用于检索、回溯和规则编译支持。

***\*4.4.4数据一致性要求\****

· 删除结构化政策时，应避免遗留失效的匹配记录引用。

· 原始政策应通过 page_url 等字段避免重复入库。

· 自动爬虫配置与运行结果应写入数据库，便于后台查看和审计。

· 画像、政策和匹配记录之间的主外键关系应保持一致。

| 聊天流式接口           | 首包返回 ≤ 1.2s              | SSE 流式推送，优先返回核心内容 |
| 政策文本编译           | 单条 ≤ 10s                   | 大模型异步调用，进度提示       |
| 文件解析（Word/PDF）   | ≤ 12s                        | 多线程提取，OCR 按需调用       |
| 风向标生成             | 单次 ≤ 50s                   | 离线预计算，增量更新           |
| 手动爬虫抓取           | 单次 ≤ 4min                  | 限流抓取，去重优化             |
| 页面加载               | 首屏 ≤ 3s                    | CDN、懒加载、资源压缩          |

 

***\*5.3 灵活性\****

系统应具备以下灵活性：

· 支持通过环境变量配置数据库地址、API 基础地址、JWT 密钥和第三方服务密钥。

· 支持在画像模型中通过扩展字段增加新的采集项，而不必重构全部数据结构。

· 支持新增政策来源、扩展爬虫数据源和调整自动爬虫参数。

· 支持新增政策条件类型与规则节点描述方式。

· 支持在前端页面中扩展新的管理页、展示页和分析页。

· 支持在不同浏览器窗口尺寸下完成基本访问和操作。

***\*6 运行需求\****

***\*6.1 用户界面\****

系统采用浏览器访问的 Web 界面，界面分为普通用户侧和管理员侧。

***\*6.1.1普通用户侧界面：\****

· 首页：用于介绍系统主线并引导进入聊天匹配。

· 聊天工作台：用于挂载画像、发起政策咨询、展示体检卡片、相关政策和风向标侧栏。

· 洞察页：用于展示政策辅助分析结果。

· 风向标页：用于查看总览、主题趋势和机会信号。

***\*6.1.2管理员侧界面：\****

· 登录页。

· 运营工作台。

· 政策管理页。

· 新增政策页。

· 编辑政策页。

· 导入与爬虫页。

· 自动爬虫配置页。

***\*6.1.3用户界面要求：\****

· 页面信息层次清晰，能够区分用户操作区、结果展示区和辅助信息区。

· 聊天与匹配结果应尽量白话化，减少直接暴露技术字段给普通用户。

· 管理后台应以表格、表单和状态提示为主，便于高频维护操作。

· 不同页面之间应具备明确导航关系。

***\*6.2 软件接口\****

系统主要软件接口如下：

***\*6.2.1\*******\*前后端接口\****

· 前端通过 HTTP/JSON 调用后端 REST API。

· API 统一前缀为 /api/v1。

· 后台受保护接口采用 Bearer Token 鉴权。

· 聊天接口采用 SSE 方式返回流式内容。

***\*6.2.2\*******\*数据库接口\****

· 后端通过 SQLAlchemy 连接 MySQL 数据库。

· 启动时可自动建表并初始化默认管理员。

***\*6.2.3\*******\*大模型接口\****

· 系统通过兼容 OpenAI 风格的接口调用 DeepSeek。

· 主要用于文本编译、聊天增强、摘要补全、风向标生成和 AI 筛选。

***\*6.2.4\**** ***\*OCR 与文件解析接口\****

· 系统支持解析 Word 和 PDF 文件。

· 对扫描版 PDF 可接入百度 OCR 进行识别。

***\*6.2.5\*******\*爬虫与外部数据接口\****

· 系统从支持的政策公开站点抓取列表页与详情页文本。

· 自动爬虫通过定时任务周期执行。

***\*6.3 硬件接口\****

本系统不依赖专用硬件设备，主要硬件需求如下：

· 服务端需具备运行 Python 后端服务、MySQL 数据库和定时任务的基础计算资源。

· 客户端需具备能够运行现代浏览器的个人计算机或笔记本电脑。

· 若启用 OCR、大模型调用和在线爬虫功能，系统运行环境需具备稳定网络连接。

· 为保存政策原文、结构化政策和运行日志，服务端需具备基本持久化存储能力。

***\*7.2 质量属性\****

***\*7.2.1可用性\****

· 系统应保证主要页面入口清晰，普通用户能够快速找到聊天匹配主工作流。

· 管理员应能够在后台完成政策库维护、导入和配置操作。

· 错误提示应尽量清晰，避免仅返回技术性异常信息。

***\*7.2.2可维护性\****

· 系统前后端采用模块化目录组织，接口、模型、服务和页面职责相对清晰。

· 数据表设计应支持后续扩展更多政策来源、规则类型和展示模块。

· 配置类参数应尽量通过环境变量或配置表管理。

***\*7.2.3安全性\****

· 后台写接口必须经过身份认证。

· 生产环境应配置独立 JWT 密钥，不得继续使用默认开发密钥。

· 管理员默认初始账号在正式部署后应及时修改。

· 对敏感配置如数据库连接串、第三方密钥应避免硬编码暴露。

***\*7.2.4可扩展性\****

· 系统应支持扩展新的政策数据源、政策主题、主体类型和画像字段。

· 系统应支持在原有结构基础上增加新的分析页面或算法模块。

· 条件树和 JSON 扩展字段设计应支持后续能力演进。

***\*7.2.5可移植性\****

· 系统采用常见 Web 技术栈，可在常规开发机或服务器上部署。

· 前端和后端可分别独立部署。

· 只要目标环境支持 Python、Node.js 和 MySQL，即可进行迁移部署。

***\*7.2.6可靠性\****

· 系统应保证核心数据如画像、政策、匹配记录和运行配置可持久化保存。

· 自动任务和外部服务异常时应尽量记录日志并返回失败信息。

· 对外部依赖不可用的情况应允许功能降级，而不是导致全部系统不可访问。
