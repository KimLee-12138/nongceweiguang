### 第一阶段：知识库基建 —— 把“死文件”变成“活规则” (后台/离线)

在用户进来之前，你们的数据库里必须是“懂规矩”的结构化数据，而不是一堆 PDF 纯文本。

1. **原文录入**：爬虫或管理员把湖北省农业农村厅的政策原文（长篇大论）抓取到 `hubei_policies_raw` 表。
2. **AI 规则编译 (DeepSeek 登场)**：系统调用 DeepSeek，把长文本政策“编译”成计算机能看懂的**JSON条件树 (`condition_tree`)**。
   - *例如*：DeepSeek 提取出规则 -> 根节点(AND) -> 子节点1(经营面积 > 50亩, 必须满足) -> 子节点2(绿色认证 == 是, 建议满足)。
3. **人工审核与入库**：管理员在后台确认 AI 生成的条件树无误后，正式存入 `policies` 结构化政策表。

### 第二阶段：用户画像挂载 —— 沉浸式的数据采集 (前台/在线)

也就是咱们刚刚讨论的多邻国式交互。

1. **首次登录拦截**：新农户登录后，系统强制弹出全屏的 `ProfileOnboarding.vue`。
2. **渐进式数据采集**：通过 20 道轻松的卡片交互题，收集用户的固定字段（如主体类型 `type`、面积 `area`）和动态扩展字段（如种植作物 `crops`，存入 `extra_data` JSON 中）。
3. **画像固化**：提交后，数据库生成唯一的 `user_profile_id`，此时农户在系统中有了明确的“数字分身”。

### 第三阶段：核心引擎匹配 —— 毫秒级的“白盒”体检 (核心业务层)

当农户进入系统，想要看看自己能申报什么政策时。

1. **触发预览**：前端携带 `user_profile_id` 向后端请求推荐政策。
2. **DFS 确定性引擎运转 (非 AI 环节)**：后端 `match_engine.py` 启动。它拿着农户的画像数据，去遍历数据库里每条政策的 JSON `condition_tree`，执行**深度优先遍历 (DFS)**。
   - *判断逻辑*：农户面积 60亩 > 政策要求 50亩 (Pass)；农户无绿色认证 != 政策要求有认证 (Fail)。
3. **生成体检报告**：匹配引擎瞬间输出一份极其准确的 `MatchSummary` 结构体，包含：`fully_matched` (是否完全匹配)、`failed_must_nodes` (卡在哪了)、`action_steps` (行动建议)。
4. **前端展示**：用户在界面上看到政策列表，以及清晰的标签（如：🟢 匹配度高 / 🔴 缺1项材料）。

### 第四阶段：对话式深度解读 —— 贴身的“AI 政策专员” (前台/DeepSeek 协同)

当农户对某个亮起红灯（有条件不满足）或感兴趣的政策点击“向 AI 咨询”时。

1. **构建超级上下文 (Prompt 组装)**：前端不仅把用户的提问（比如：“我这个绿色认证去哪办啊？”）发给后端，后端还会**悄悄把三个核心数据打包**：
   - 农户当前的画像 (`user_profiles`)
   - 该政策的原文与摘要 (`policies`)
   - 刚刚引擎算出来的匹配失败报告 (`MatchSummary`)
2. **向 DeepSeek 发起召唤**：后端把这个“超级上下文”喂给 DeepSeek，并给它设定 System Prompt：“你是一个专业的农业政策解读员，请结合用户的具体情况和匹配失败原因，用大白话为他解答，严禁胡编乱造”。
3. **SSE 流式响应**：DeepSeek 开始生成回复。后端通过 Server-Sent Events (SSE) 技术，像打字机一样把解答一个字一个字地推送到用户的聊天窗口中。
   - *AI 的回答示例*：“王大哥，您这个家庭农场面积是够的，但卡在了‘绿色认证’上。根据政策第3条，您需要去当地农业局补办这个证，您可以先准备好土地流转合同...”
4. **会话沉淀**：这次高质量的对话被保存在 `chat_messages` 表中，随时可以回溯。
