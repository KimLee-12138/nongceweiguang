## 1 引言

### 1.1 编写目的

本说明书用于描述“农策微光”系统的业务目标、功能需求、数据需求、性能需求、运行需求及其他质量要求，为课程项目开发、阶段验收、演示答辩、后续维护与功能扩展提供统一依据。

本说明书面向以下读者：

- 项目组成员，用于统一需求理解与开发边界。
- 指导教师与课程评审人员，用于了解系统的业务定位、功能完成度和实现范围。
- 后续维护人员，用于快速理解系统结构、接口关系与核心数据对象。

### 1.2 背景

在农业政策申报场景中，用户通常面临以下问题：政策来源分散、原文专业性强、适用条件复杂、个人或主体信息与政策条款难以快速对齐。传统方式往往依赖人工查阅政策文件、逐条比对条件，效率较低，且容易遗漏关键条款。

“农策微光”面向这一痛点，尝试构建一个农业政策智能匹配与申报辅助平台。系统围绕“农户画像挂载、政策规则化、确定性匹配、白话解读、后台治理”五条主线展开，将政策原文转化为结构化条件树，并结合农户或农业经营主体画像输出更可解释的匹配结果和行动建议。

本项目当前采用前后端分离架构实现，前端负责页面交互与结果展示，后端负责数据存储、政策管理、规则匹配、对话接口、文件解析、爬虫同步和风向标生成等能力。

## 2 任务概述

### 2.1 项目概述

#### 2.1.1 项目来源及背景

本项目来源于软件工程课程团队项目实践，目标是围绕真实农业政策服务场景，设计并实现一套具有实际演示价值的农业政策智能匹配与辅助申报系统。

从业务背景看，农业经营主体在寻找适合自身情况的扶持政策时，往往需要面对政策原文长、门槛条款多、适用范围复杂、资料准备要求不明确等问题。系统需要提供一种比传统关键词搜索更直观、更可解释的解决方案，将“政策可看见”进一步转化为“政策可理解、可判断、可行动”。

#### 2.1.2 项目目标

本项目的主要目标如下：

- 建立农业政策管理与持续扩充机制，支持手工录入、文件导入和爬虫抓取。
- 将政策原文编译为结构化规则，形成可计算的政策条件树。
- 支持用户建立农户画像或主体画像，并基于画像进行政策匹配。
- 为用户提供白话解读、缺口分析、行动建议和相关政策推荐。
- 为管理员提供登录鉴权、政策维护、导入治理和自动爬虫配置能力。
- 为项目演示提供政策洞察和政策风向标等辅助展示能力。

#### 2.1.3 系统概述

“农策微光”是一个面向农业政策申报场景的 Web 系统，采用前后端分离方式实现。

- 前端采用 Vue 3 + Vite，负责首页引导、聊天工作台、洞察展示页和后台管理页。
- 后端采用 FastAPI + SQLAlchemy，负责认证、画像管理、政策管理、匹配计算、聊天流式接口、风向标接口、文件解析、爬虫与自动任务等。
- 数据层采用 MySQL，保存管理员、农户画像、结构化政策、匹配记录、原始政策、系统配置和风向标报告等数据。
- AI 能力用于离线规则编译和结果增强，而非将所有判断完全交给大模型。

系统定位统一为：农业政策智能匹配与申报辅助平台。

### 2.2 用户特点

本系统主要面向以下两类用户：

1. 普通用户
   包括农户个人、家庭农场、农民合作社、农业企业等农业经营主体。其特点是：

- 了解政策需求强，但缺乏对政策条文的专业理解能力。
- 更关注“我能不能申报”“我还缺什么条件”“接下来该做什么”。
- 期望使用门槛低，偏好可视化、白话化、对话式交互。

2. 管理用户
   即系统管理员或政策库维护人员。其特点是：

- 需要维护政策知识库，保证政策条目持续入库和更新。
- 需要管理自动爬虫与文件导入等后台流程。
- 需要对外展示系统运行成果和平台数据概况。

### 2.3 假定和约束

本系统在当前阶段的主要假定和约束如下：

- 假定系统运行于具备网络连接的常规 Web 环境，用户通过浏览器访问。
- 假定系统部署时可连接 MySQL 数据库。
- 假定第三方模型服务在需要时可用，否则部分 AI 相关功能仅返回提示或降级结果。
- 假定 OCR 服务在处理扫描版 PDF 时已完成正确配置，否则仅支持可直接提取文字的文件。
- 当前系统主线以“画像挂载 + 规则匹配 + 白话解读 + 管理治理”为准，不以旧版 ROI 叙事作为核心能力。
- 当前原始政策来源以湖北省农业农村厅政府信息公开数据为主，区域覆盖范围仍有限。
- 当前宏观洞察页面主要承担辅助展示功能，其分析深度受现有数据底座限制。
- 当前测试、部署与安全配置更适合课程项目与演示环境，正式生产化落地仍需进一步完善。

- 手动政策爬虫抓取与导入。

- 政策风向标手动生成与查看。

### 3.2 功能描述

#### 3.2.1 农户画像管理

功能说明：系统应支持创建、查看、修改农户画像或经营主体画像，作为后续政策匹配的输入基础。

输入：

- 主体名称或用户名称。
- 主体类型。
- 经营面积。
- 绿色认证情况。
- 灌溉方式。
- 动态扩展字段，如作物类型、经营年限、保险情况等。

处理：

- 系统对基础字段进行结构化保存。
- 对非固定字段采用扩展键值对方式保存。

输出：

- 返回画像主键 ID。
- 返回画像详情。
- 返回画像历史匹配记录。

#### 3.2.2 政策智能匹配

功能说明：系统应支持基于农户画像与结构化政策条件树进行规则匹配，判断用户是否满足政策要求，并输出结构化匹配结果。

输入：

- 已保存的农户画像 ID。
- 待匹配的政策 ID。

处理：

- 根据政策条件树进行本地确定性匹配。
- 区分必须条件和建议条件。
- 汇总失败节点、原因说明、风险提示和行动建议。

输出：

- 是否完全匹配。
- 未通过的必须条件和建议条件。
- 全部条件节点评估结果。
- 白话解释、行动清单和风险预警。

#### 3.2.3 对话式政策解读

功能说明：系统应支持用户通过聊天工作台发起政策咨询，系统以流式方式返回自然语言回答，同时可附带政策体检结果和解读结果。

输入：

- 用户消息内容。
- 可选的画像 ID。
- 可选的政策 ID 或政策原文 ID。
- 聊天模式，如匹配模式、解读模式。

处理：

- 根据用户问题检索相关政策。
- 在可用情况下结合画像和政策规则生成政策体检结果。
- 通过 SSE 持续推送聊天内容。

输出：

- 流式文本回答。
- 政策体检结果对象。
- 政策解释或引导性回答。

#### 3.2.4 相关推荐政策

功能说明：系统应根据当前画像，对政策库进行匹配预览，返回适合当前主体的一组政策候选。

输入：

- 画像 ID。
- 偏移量和返回条数。

处理：

- 对政策库进行批量匹配预览。
- 过滤公示类和不适合作为申报政策的条目。
- 按匹配程度排序。

输出：

- 政策 ID、标题、来源、摘要、原文链接。
- 是否完全匹配。
- 是否包含风险提醒。
- 是否存在行动建议。

#### 3.2.5 政策洞察展示

功能说明：系统应提供政策洞察页，用于展示政策数量、来源分布、目标受众和趋势分析等辅助信息。

输入：

- 无强制业务输入，页面加载时按接口请求快照数据。

处理：

- 后端聚合并缓存洞察数据快照。
- 前端将结果渲染为图表与指标卡片。

输出：

- KPI 汇总。
- 趋势分析数据。
- 来源分布数据。
- 受众分布数据。

#### 3.2.6 管理员登录与鉴权

功能说明：系统应支持管理员账号登录，并通过 JWT 令牌保护后台接口。

输入：

- 用户名。
- 密码。

处理：

- 校验管理员账号与密码。
- 登录成功后签发访问令牌。
- 受保护接口通过 Bearer Token 校验身份。

输出：

- 访问令牌。
- 当前管理员身份信息。

#### 3.2.7 结构化政策管理

功能说明：系统应支持管理员对结构化政策进行新增、列表查看、详情查看、编辑和删除。

输入：

- 政策标题、来源、版本、摘要、原文链接。
- 结构化条件树。

处理：

- 将政策以结构化对象形式保存到数据库。
- 支持分页查询和按 ID 查询。
- 删除政策时同步清理相关匹配记录引用。

输出：

- 政策列表。
- 单条政策详情。
- 新增、更新和删除结果。

#### 3.2.8 政策文本编译

功能说明：系统应支持将原始政策文本提交给规则编译服务，生成结构化政策对象。

输入：

- 政策标题。
- 政策来源。
- 原始政策文本。

处理：

- 调用大模型对文本进行结构化解析。
- 输出符合系统规则的条件树和摘要信息。

输出：

- 编译后的结构化政策对象。
- 编译失败时返回错误提示。

#### 3.2.9 文件解析与批量导入

功能说明：系统应支持管理员上传 Word 或 PDF 文件，提取正文后选择条目批量导入结构化政策库。

输入：

- `.docx` 或 `.pdf` 文件。
- 选择导入的政策条目。

处理：

- 提取文件正文。
- 为扫描版 PDF 在需要时调用 OCR。
- 将提取结果返回前端供人工勾选。
- 对选中条目执行批量编译和入库。

输出：

- 文件解析结果列表。
- 批量导入成功与失败数量。
- 失败明细。

#### 3.2.10 手动政策爬虫导入

功能说明：系统应支持从已支持的数据源手动抓取政策原文，供管理员复核后导入。

输入：

- 数据源标识。
- 抓取页数等参数。

处理：

- 爬取列表页与详情页。
- 去重、抽取正文、整理原文链接。
- 返回新增候选条目。

输出：

- 本次抓取到的政策候选列表。
- 原文标题、来源、正文和链接。

#### 3.2.11 自动爬虫配置与运行记录

功能说明：系统应支持管理员配置自动爬虫的开关、数据源、抓取间隔和最大抓取页数，并查看最近运行情况。

输入：

- 是否启用自动爬虫。
- 数据源列表。
- 间隔小时数。
- 每次抓取页数上限。

处理：

- 将配置持久化到系统配置表。
- 定时任务按周期触发自动抓取与导入。
- 记录运行结果。

输出：

- 当前配置内容。
- 最近一次运行状态。
- 抓取、筛选、导入、失败数量统计。

#### 3.2.12 政策风向标与智库词典

功能说明：系统应支持生成和展示政策风向标周报及智库词典，辅助用户理解近期政策趋势。

输入：

- 定时任务或管理员手动触发。
- 后端结构化政策信号与近期政策数据。

处理：

- 聚合政策信号。
- 调用大模型生成报告摘要与正文。
- 生成同周期术语解释列表。

输出：

- 总览、主题趋势、机会信号。
- 最新风向标报告列表和详情。
- 智库词典条目列表。

## 4 数据需求

### 4.1 静态数据

系统的静态数据主要包括：

- 系统预置管理员账号初始化规则。
- 自动爬虫默认配置，如默认数据源、抓取间隔、最大页数。
- 字段标签映射与前端展示文案。
- 风向标分类枚举，如扶持方向、产业走向、区域重点、申报要点。
- 政策来源标识与映射规则。
- 前端固定页面路由与后台菜单结构。

静态数据特点：

- 更新频率较低。
- 多由系统配置或代码常量提供。
- 用于保障界面与业务逻辑的一致性。

### 4.2 动态数据

系统运行过程中产生和更新的动态数据主要包括：

- 农户画像数据。
- 结构化政策数据。
- 政策匹配记录数据。
- 原始政策原文数据。
- 管理员登录后的令牌与鉴权上下文。
- 政策洞察快照数据。
- 政策风向标报告和智库词典数据。
- 自动爬虫运行记录。
- 文件解析结果和批量导入结果。
- 聊天流式会话中的响应内容。

动态数据特点：

- 更新频率高。
- 部分依赖外部数据源或第三方服务。
- 需要通过数据库持久化或缓存机制保证使用体验。

### 4.3 数据字典

| 数据实体 | 主要字段 | 说明 |
| --- | --- | --- |
| 管理员 `admin_users` | `id`、`username`、`password_hash`、`created_at` | 用于后台登录与鉴权。 |
| 农户画像 `user_profiles` | `id`、`external_id`、`name`、`type`、`area`、`green_cert`、`irrigation`、`extra_data` | 用于保存农户或经营主体画像信息。 |

| 原始政策 `hubei_policies_raw` | `id`、`title`、`issuer`、`doc_no`、`publish_date`、`column`、`file_category`、`topic_category`、`page_url`、`attachment_urls`、`html_text`、`attachment_text`、`full_text` | 用于保存爬虫抓取和解析后的原始政策文本。 |

### 4.4 数据库描述

1. 数据库类型

系统采用 MySQL 作为关系型数据库，后端通过 SQLAlchemy ORM 对数据表进行映射与访问。

2. 主要表之间的关系

- `user_profiles` 与 `match_records` 为一对多关系，即一个画像可对应多条匹配记录。
- `policies` 与 `match_records` 为一对多关系，即一条政策可被多个画像匹配。
- `system_config` 采用键值对方式存储全局配置。
- `compass_reports` 与 `compass_glossary` 通过发布时间形成逻辑上的周期对应关系。

3. 主要设计特点

- `policies.condition_tree` 使用 JSON 存储完整条件树，便于直接参与规则匹配。
- `user_profiles.extra_data` 使用 JSON 存储动态扩展字段，便于适应不同政策场景。
- `match_records.match_detail` 使用 JSON 存储完整匹配结果，便于后续展示与复盘。
- `hubei_policies_raw.full_text` 存储原文全文，用于检索、回溯和规则编译支持。

4. 数据一致性要求

- 删除结构化政策时，应避免遗留失效的匹配记录引用。
- 原始政策应通过 `page_url` 等字段避免重复入库。
- 自动爬虫配置与运行结果应写入数据库，便于后台查看和审计。
- 画像、政策和匹配记录之间的主外键关系应保持一致。

## 5 性能需求

### 5.1 数据精度

系统对数据精度的要求如下：

- 农户画像数据应尽量准确反映用户当前经营主体情况，尤其是主体类型、面积、认证和灌溉等核心字段。
- 结构化政策中的条件树应与原始政策条款保持一致，避免出现核心条件丢失、误判或逻辑关系错误。
- 文件解析与 OCR 提取的文本应尽可能保留政策原文的关键信息，若识别质量不足，应允许管理员复核后再导入。
- 匹配结果准确性依赖于政策规则质量和画像完整度。当画像字段缺失或规则不完整时，系统应优先给出提示，而不是假定用户满足条件。
- 洞察和风向标展示应尽量基于真实数据生成，不应在用户可见结果中误导为已验证的精确统计。

### 5.2 时间特性

系统在课程演示和一般 Web 环境下应满足以下时间特性要求：

- 普通查询类接口，如登录校验、政策列表、画像列表、详情获取，应在可接受的交互时间内返回。
- 画像创建、更新、政策保存、配置保存等写操作，应在短时间内完成并给出明确反馈。
- 聊天接口应尽快返回流式首包，以降低用户等待感。
- 政策编译、文件解析、OCR 识别、风向标生成等依赖外部服务的功能允许耗时较长，但应提供状态提示或结果反馈。
- 自动爬虫与周报生成属于后台任务，不应阻塞普通用户的页面访问。

为便于验收，可采用以下保守目标作为参考：

- 一般 CRUD 接口：正常情况下应在 3 秒内返回结果。
- 分页列表和详情接口：正常情况下应在 5 秒内返回结果。
- 聊天流式接口：应在合理等待时间内开始推送内容。
- 编译、解析、OCR 和风向标生成接口：允许较长耗时，但应具备超时提示或失败提示。

### 5.3 灵活性

系统应具备以下灵活性：

- 支持通过环境变量配置数据库地址、API 基础地址、JWT 密钥和第三方服务密钥。
- 支持在画像模型中通过扩展字段增加新的采集项，而不必重构全部数据结构。
- 支持新增政策来源、扩展爬虫数据源和调整自动爬虫参数。
- 支持新增政策条件类型与规则节点描述方式。
- 支持在前端页面中扩展新的管理页、展示页和分析页。
- 支持在不同浏览器窗口尺寸下完成基本访问和操作。

## 6 运行需求

### 6.1 用户界面

系统采用浏览器访问的 Web 界面，界面分为普通用户侧和管理员侧。

普通用户侧界面包括：

- 首页：用于介绍系统主线并引导进入聊天匹配。
- 聊天工作台：用于挂载画像、发起政策咨询、展示体检卡片、相关政策和风向标侧栏。
- 洞察页：用于展示政策辅助分析结果。
- 风向标页：用于查看总览、主题趋势和机会信号。

管理员侧界面包括：

- 登录页。
- 运营工作台。
- 政策管理页。
- 新增政策页。
- 编辑政策页。
- 导入与爬虫页。
- 自动爬虫配置页。

用户界面应满足以下要求：

- 页面信息层次清晰，能够区分用户操作区、结果展示区和辅助信息区。
- 聊天与匹配结果应尽量白话化，减少直接暴露技术字段给普通用户。
- 管理后台应以表格、表单和状态提示为主，便于高频维护操作。
- 不同页面之间应具备明确导航关系。

### 6.2 软件接口

系统主要软件接口如下：

1. 前后端接口

- 前端通过 HTTP/JSON 调用后端 REST API。
- API 统一前缀为 `/api/v1`。
- 后台受保护接口采用 Bearer Token 鉴权。
- 聊天接口采用 SSE 方式返回流式内容。

2. 数据库接口

- 后端通过 SQLAlchemy 连接 MySQL 数据库。
- 启动时可自动建表并初始化默认管理员。

3. 大模型接口

- 系统通过兼容 OpenAI 风格的接口调用 DeepSeek。
- 主要用于文本编译、聊天增强、摘要补全、风向标生成和 AI 筛选。

4. OCR 与文件解析接口

- 系统支持解析 Word 和 PDF 文件。
- 对扫描版 PDF 可接入百度 OCR 进行识别。

5. 爬虫与外部数据接口

- 系统从支持的政策公开站点抓取列表页与详情页文本。
- 自动爬虫通过定时任务周期执行。

### 6.3 硬件接口

本系统不依赖专用硬件设备，主要硬件需求如下：

- 服务端需具备运行 Python 后端服务、MySQL 数据库和定时任务的基础计算资源。
- 客户端需具备能够运行现代浏览器的个人计算机或笔记本电脑。
- 若启用 OCR、大模型调用和在线爬虫功能，系统运行环境需具备稳定网络连接。
- 为保存政策原文、结构化政策和运行日志，服务端需具备基本持久化存储能力。

### 7.2 质量属性

1. 可用性

- 系统应保证主要页面入口清晰，普通用户能够快速找到聊天匹配主工作流。
- 管理员应能够在后台完成政策库维护、导入和配置操作。
- 错误提示应尽量清晰，避免仅返回技术性异常信息。

2. 可维护性

- 系统前后端采用模块化目录组织，接口、模型、服务和页面职责相对清晰。
- 数据表设计应支持后续扩展更多政策来源、规则类型和展示模块。
- 配置类参数应尽量通过环境变量或配置表管理。

3. 安全性

- 后台写接口必须经过身份认证。
- 生产环境应配置独立 JWT 密钥，不得继续使用默认开发密钥。
- 管理员默认初始账号在正式部署后应及时修改。
- 对敏感配置如数据库连接串、第三方密钥应避免硬编码暴露。

4. 可扩展性

- 系统应支持扩展新的政策数据源、政策主题、主体类型和画像字段。
- 系统应支持在原有结构基础上增加新的分析页面或算法模块。
- 条件树和 JSON 扩展字段设计应支持后续能力演进。

5. 可移植性

- 系统采用常见 Web 技术栈，可在常规开发机或服务器上部署。
- 前端和后端可分别独立部署。
- 只要目标环境支持 Python、Node.js 和 MySQL，即可进行迁移部署。

6. 可靠性

- 系统应保证核心数据如画像、政策、匹配记录和运行配置可持久化保存。
- 自动任务和外部服务异常时应尽量记录日志并返回失败信息。
- 对外部依赖不可用的情况应允许功能降级，而不是导致全部系统不可访问。
