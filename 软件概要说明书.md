# ***\*1 引言\****

## ***\*1.1 编写目的\****

本说明书用于从总体上描述“农策微光”软件系统的设计方案，明确系统定位、模块组成、层次关系、关键接口、数据实体、运行方式和非功能约束，为后续开发协作、课程答辩、维护交接和版本演进提供统一依据。

与需求规格说明书相比，概要设计说明书强调“系统如何被组织与实现”。因此，本文件不再单纯罗列需求，而是围绕架构层次、模块职责、关键流程、数据库模型和运行机制进行展开，并尽量将仓库内已经落地的实现状态转写为可读的工程文档。

## ***\*1.2 项目背景\****

农业扶持政策的公开文本通常分散在政府站点、栏目公告和附件文件中，文本风格高度行政化，适用对象、申报条件、材料要求和时效限制往往交织在同一份长文中。普通农户、家庭农场、合作社和涉农企业即使具备政策需求，也难以在短时间内准确判断“自己是否适用、差什么条件、接下来该怎么准备”。

“农策微光”面向上述痛点，构建了一个集政策原文采集、结构化规则编译、农户画像建模、白盒匹配、对话式解读、审核治理和趋势洞察于一体的 Web 平台。系统不是简单做关键字搜索，而是把政策文本沉淀为可计算的条件树，再结合用户画像做本地确定性判断，最终输出更容易理解的资格结论、缺口说明和行动建议。

## ***\*1.3 术语和缩略语\****

| **术语/缩略语** | **说明**                                                     |

# ***\*2 总体设计\****

## ***\*2.1 建设目标\****

· 构建可持续更新的农业政策知识底座，支持手动录入、文件解析、人工抓取和自动爬虫多种入库来源。

· 将政策原文编译成可计算的结构化规则树，使匹配判断尽量由白盒逻辑完成，而不是完全交由大模型自由生成。

· 为普通用户提供画像建模、政策匹配、对话解读和最近会话沉淀能力，降低政策理解门槛。

· 为管理员提供审核、任务中心、自动化配置和趋势报告生成等治理能力，确保政策库可维护、可追踪、可复查。

· 在课程项目场景下兼顾展示效果与工程完整性，使系统既能答辩展示，也能支持继续扩展。

## ***\*2.2 用户角色与典型场景\****

| **角色**          | **主要关注点**                                       | **典型入口**                       | **设计要点**                                     |
| ----------------- | ---------------------------------------------------- | ---------------------------------- | ------------------------------------------------ |
| 普通用户          | 我能否申报、差哪些条件、怎样理解政策文本             | /login、/register、/chat、/compass | 强调画像输入简洁、对话交互自然、结果可解释。     |
| 管理员            | 如何维护政策库、审核 AI 草稿、查看后台任务与失败原因 | /admin/login、/admin/*             | 强调治理闭环、任务可观测、可重试和审计留痕。     |
| 课程评审/演示观众 | 系统是否有真实数据来源、真实流程和趋势分析能力       | /insights、/compass、/chat         | 强调大屏展示、风向标报告和端到端业务链路完整度。 |

 

## ***\*2.3 设计原则\****

· 事实优先原则：设计说明以仓库内可核验的代码、接口和数据模型为准，不主动扩写未落地功能。

· 规则优先原则：政策匹配核心结论由本地确定性规则得出，AI 主要承担离线编译、解释增强和文稿生成。

· 治理闭环原则：原文采集、审核入队、草稿编辑、审批入库、任务追踪和异常重试形成完整链路。

· 前后端分离原则：前端专注交互组织和可视化呈现，后端承载业务规则、存储、调度和第三方能力接入。

· 异步削峰原则：抓取、文件解析、批量导入、批量补摘要和风向标生成等高成本能力统一纳入后台作业中心。

· 可降级运行原则：当大模型、OCR 或部分 schema 能力不可用时，系统尽量以明确提示和局部降级代替整体不可用。

# ***\*3 系统架构设计\****

## ***\*3.1 分层架构\****

| **层次**         | **职责说明**                                                 | **典型内容**                                                 |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 表现层           | 负责页面布局、交互编排、状态展示和图表渲染，不直接承载核心业务规则。 | 首页、聊天工作台、风向标大屏、管理端、任务中心。             |
| 接口层           | 负责 URL 路由、鉴权依赖、请求校验、响应模型和错误码转换。    | auth_routes、profile_routes、policy_routes、chat_routes、admin_ops_routes。 |
| 应用服务层       | 封装具体业务流程，组织数据访问、第三方调用和规则判断。       | match_engine、review_queue_service、file_parse_service、admin_operation_service。 |
| 数据与基础设施层 | 负责 ORM、数据库会话、运行时 schema 校验、日志、安全、限流、调度和第三方客户端。 | models、db、runtime_schema、core、scheduler、ai、telemetry。 |

 

这种分层方式使控制器保持轻量，复杂逻辑沉入服务层，数据对象由 Schema 与 ORM 分别承载，从而降低接口代码、业务代码和持久化代码之间的耦合度。

## ***\*3.2 前端架构设计\****

前端采用 Vue 3 组合式 API 构建单页应用，使用 Vue Router 管理页面级路由，使用 Element Plus 提供基础交互组件，使用 ECharts 负责趋势类可视化。前端不保存 JWT 到 localStorage，而是依赖 HttpOnly Cookie 与 `me` 接口进行登录态恢复，既降低了前端凭据暴露面，也与后端的 refresh 机制保持一致。

· 页面层：`HomePage.vue`、`ChatPage.vue`、`PolicyCompassView.vue`、`BusinessInsightView.vue` 与 `views/admin/*` 承担页面组织职责。

· 组件层：聊天区、政策体检卡片、推荐列表、报告抽屉、任务卡片等通用组件拆分在 `components/chat` 与 `components/common`。

· 服务层：`src/api/client.js` 统一封装接口访问、自动刷新和会话标记；`authSession.js` 管理用户/管理员会话探测。

· 组合式逻辑：`useChatSessions`、`usePersistentChatStream`、`useProfileEditor` 等 composable 抽离了会话、流式消息和画像编辑状态。

· 展示策略：聊天工作台、管理端和大屏页面分别采用不同的信息密度和视觉重点，以适配普通用户、管理员与演示场景。

## ***\*3.3 后端架构设计\****

后端以 `create_app()` 为入口构建 FastAPI 应用，统一挂载认证、画像、匹配、聊天、政策、风向标、洞察和后台作业路由。系统通过 SQLAlchemy ORM 管理数据实体，依赖 `pydantic` Schema 定义请求与响应模型，利用 `runtime_schema` 在启动阶段判断数据库结构是否满足当前代码要求。

· API 层只保留路由绑定、参数解析、权限约束和错误处理，不直接拼装复杂业务。

· 服务层负责业务编排，例如匹配结果计算、原文抓取、审核任务创建、文件解析和 telemetry 聚合。

· 模型层负责数据持久化结构，涵盖账号会话、画像政策、聊天消息、审核任务、后台作业和观测事件。

· 调度层在独立进程运行，通过数据库命名锁避免多实例重复执行周期任务。

· AI 与 OCR 调用被封装在服务层或专用适配模块中，使其可被替换、禁用或降级。

## ***\*3.4 核心业务流程\****

### ***\*3.4.1 农户画像与政策匹配流程\****

1. 普通用户注册或登录后进入聊天工作台，可新增或编辑自己的农户画像。
2. 画像数据保存到 `user_profiles`，固定字段记录在显式列中，可扩展信息进入 `extra_data`。
3. 当用户发起某条政策的匹配请求时，后端读取结构化政策的 `condition_tree` 和当前画像。
4. 匹配引擎以 DFS 方式遍历条件树，对每个节点判断字段值、运算符和阈值是否满足，并区分 must/should 条件。
5. 系统生成 `MatchSummary` 结构，输出整体是否匹配、失败节点、行动建议和风险提示，并落库到 `match_records`。
6. 若存在失败节点，系统可额外调用 AI 对说明和行动建议进行润色，但不会替代白盒判断结果。

### ***\*3.4.2 政策原文导入与审核入库流程\****

7. 管理员通过文件上传、手动抓取或自动爬虫获得政策原文候选集。
8. 原文候选首先进入后台作业或预览列表，而不是直接写入结构化政策库。
9. 管理员可将选中的原文创建为审核任务，审核任务会记录原文、来源、草稿状态和 AI 状态。
10. 系统尝试调用规则编译能力生成结构化条件树草稿，同时生成 AI 摘要、分类、风险点和建议。
11. 管理员在审核台修改草稿、保存版本并作出通过或驳回决策。
12. 审核通过后，系统把草稿写入 `policies` 表，并将审核任务标记为 approved；驳回则保留原因和时间。

### ***\*3.4.3 聊天工作台与 SSE 流程\****

13. 前端在 `/chat` 页面管理最近会话、当前画像、输入草稿和当前引用政策。
14. 发送消息时，后端先确保会话存在，并为本次交互创建一对用户/助手消息记录。
15. 系统根据模式和上下文选择是否注入画像摘要、政策摘要、原文片段或检索结果。
16. 聊天回复通过 SSE 分块返回，消息流中除了 `content` 文本，还可附带 `policy_report`、`interpretation_report`、`policy_context` 等结构化事件。
17. 流式过程中后端持续更新助手消息内容和 `render_payload_json`，保障刷新后仍能恢复最近上下文。
18. 结束后系统记录 chat_stream telemetry，包括首 token 延迟、输出字符数、异常原因等。

### ***\*3.4.4 自动爬虫与后台作业流程\****

19. Scheduler 按配置周期触发自动爬虫入队与后台作业消费。
20. 自动爬虫配置保存在 `system_config` 中，包括开关、数据源、间隔小时数和每源最大抓取页数。
21. 当需要抓取、解析、批量导入或生成周报时，系统先创建 `admin_operation_runs` 记录，而不是同步执行重负载逻辑。
22. 后台消费者循环领取 pending 任务，逐项执行并写入 `admin_operation_run_items` 和 `telemetry_events`。
23. 任务中心统一展示任务状态、失败原因、OCR 成本、结果 JSON 和可重试入口。
24. 失败或部分失败的任务可创建新的 retry run，并在 telemetry 中保留重试来源关系。

### ***\*3.4.5 风向标生成流程\****

25. 系统先基于真实数据库样本计算结构化信号，包括主题趋势、主体机会和准备事项。
26. 在结构化信号充分的前提下，再调用大模型生成四类周报和术语词典。
27. 生成结果分别写入 `compass_reports` 与 `compass_glossary`，并保留最近两期数据。
28. 前端风向标页面同时展示结构化图表、报告卡片和术语解释，而不是只显示 AI 生成文案。
29. 当大模型不可用时，结构化趋势图和机会信号仍可单独工作，报告部分则返回空列表或默认提示。

## ***\*4.1 认证与会话管理模块\****

| **维度** | **说明**                                                     |

结构化政策是系统的规则中枢，普通用户侧的推荐、匹配、聊天体检与趋势分析都依赖该表。

## ***\*4.6 审核治理模块\****

# ***\*6 数据设计\****

## ***\*6.1 数据实体关系概述\****

系统数据可以分为四类：账号与会话数据、业务主数据、审核与任务治理数据、对话与趋势展示数据。整体关系上，普通用户拥有自己的画像和聊天会话；画像与结构化政策之间通过匹配记录关联；政策原文经审核任务流转后生成正式政策；后台作业和 telemetry 贯穿文件解析、抓取、批量导入、自动爬虫与风向标生成全过程。

· `end_users` 与 `user_profiles`、`chat_conversations` 为一对多关系。

· `chat_conversations` 与 `chat_messages` 为一对多关系，消息可引用画像快照和政策引用信息。

· `user_profiles` 与 `match_records` 为一对多关系，`policies` 与 `match_records` 也是一对多关系。

· `policy_review_tasks` 审批通过后写入 `policies`，并通过 `approved_policy_id` 建立逻辑关联。

· `admin_operation_runs` 与 `admin_operation_run_items`、`telemetry_events` 为一对多关系。

## ***\*7.5 异常处理与降级策略\****

· 运行时 schema 校验支持 fatal 与 degraded 两级结果：fatal 直接阻止启动，degraded 允许服务继续运行但屏蔽相关能力。

· 当 `telemetry_events` 表缺失时，任务中心与部分观测接口会降级，但认证、聊天和后台作业尽量不因观测写入失败而整体中断。

· 当 `DEEPSEEK_API_KEY` 缺失时，匹配引擎、画像管理、基础政策查询等白盒能力仍可工作，但规则编译、聊天增强、AI 审核建议和风向标文稿会退化为提示信息或空结果。

· 当 OCR 未配置时，系统仍支持直接可提取文本的 DOCX/PDF，但扫描版 PDF 解析效果会明显下降。

· 高成本接口统一采用后台作业模型和限流约束，避免阻塞 Web 线程或因误操作造成资源占满。

# ***\*8 非功能设计\****

## ***\*8.1 安全设计\****

· 使用 HttpOnly Cookie 持有 access/refresh token，前端不将 JWT 保存到 localStorage。

· 管理员与普通用户使用不同 Cookie 名称和独立路由前缀，降低身份混淆风险。

· 刷新流程支持 refresh token 轮换与重放检测，异常时清理 Cookie 并使当前会话失效。

· 生产环境强制要求非默认 `JWT_SECRET_KEY` 与显式 `CORS_ALLOW_ORIGINS`，否则后端拒绝启动。

· 系统中间件默认添加 `X-Content-Type-Options`、`Referrer-Policy` 和 `Content-Security-Policy` 等安全响应头。

· 登录接口和高成本接口实施 IP 与主体维度限流；普通用户不能访问他人画像和会话，管理接口统一要求管理员身份。

## ***\*8.2 性能与资源控制\****

· 同步文件预览限制为较小文件和较少页数；超限请求建议转后台解析，避免单次请求耗时过长。

· 聊天以 SSE 流式返回文本，同时分段持久化消息，降低用户等待体感并提高刷新恢复能力。

· 洞察大屏接口增加 300 秒内存缓存，减少重复统计查询压力。

· 后台作业拆分为 run 与 item 两级结构，便于分批执行、失败重试和结果恢复。

· 调度器消费者按 10 秒轮询 pending 作业，能够在保持简单实现的同时承担中低频异步任务。

## ***\*8.3 可维护性设计\****

· 前端通过 composable 与统一 API 客户端收敛会话、聊天流和画像编辑逻辑，减少页面级重复代码。

· 后端 API、Schema、Service、Model 分层清晰，便于定位职责和扩展测试。

· 后台作业、审核任务和 telemetry 使用统一抽象，降低新增异步能力时的设计成本。

· Alembic 迁移、环境变量样例和 README 使部署过程具备基本可复现性。
